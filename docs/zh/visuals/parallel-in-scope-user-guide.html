它适合什么问题

01 / WHY

当你有一组相互独立、需要并行执行的 IO 或计算任务，并且希望“出了问题就及时收手”，它会比手写线程池编排更简单。

↗

批量远程调用

同时请求多个接口、查询多个数据源；每个任务都有统一超时，某个失败后其余任务会尽快停止。

◌

批量计算

把 CPU 任务分到有限的并行度，避免一次性把大量工作塞进队列。

⌁

嵌套业务流程

父任务取消时，子任务会收到同一个取消信号；请求级上下文也能跨线程保留。

不适合：需要响应式链式编排、内置重试策略，或希望失败后继续收集全部结果的场景。这里选择的是清晰的 fail-fast 语义。

5 分钟接入

02 / START

① 添加依赖

<dependency> <groupId>io.github.huatalk</groupId> <artifactId>parallel-in-scope</artifactId> <version>0.2.0</version> </dependency>

② 注册线程池

应用启动时注册一次。线程池名称会成为后续调用的稳定标识。

③ 并行处理列表

ParOptions options = ParOptions.ioTask("fetchData") .parallelism(5) .timeout(3000) .build(); AsyncBatchResult<String> result = par.map( "io-pool", urls, url -> httpClient.fetch(url), options );

返回的是一组 Future。你可以逐个读取，也可以通过 result.report() 查看批次状态。

你需要知道的运行行为

03 / BEHAVIOR
你配置的内容框架会做什么你得到的结果
parallelism = 5先运行 5 个；每完成一个才补下一个线程池不会被整批任务淹没
timeout = 3000超时后取消批次，并停止后续提交调用方能更快结束等待
任务抛出异常触发同批任务的 fail-fast 取消错误尽快暴露，不继续浪费资源
父任务被取消取消信号传到嵌套任务不需要手工层层传递 token
CPU_BOUND倾向拒绝排队，必要时回到调用线程执行减少计算任务排队造成的延迟

取消是协作式的

框架不会强行终止 Java 线程。阻塞操作响应 interrupt；长时间 CPU 循环请在业务代码中调用检查点。

Checkpoints.checkpoint("compute", true);

失败策略是显式的

如果某个子任务允许失败，请在任务函数内部捕获异常并返回业务结果。框架本身保持 fail-fast，不替你猜测容错规则。

常见问题

04 / FAQ
我应该选 IO_BOUND 还是 CPU_BOUND？

等待网络、数据库、文件时选 IO_BOUND；大量计算、压缩、解析时选 CPU_BOUND。CPU 任务通常应把并行度设为 CPU 核数附近。

调用 map 后任务已经执行完了吗？

不一定。map 快速返回 AsyncBatchResult，真正的任务仍在执行。通过 Future.get、回调或 report() 观察完成状态。

超时后正在运行的任务一定会立刻停止吗？

不一定。取消是协作式的；阻塞操作通常能响应中断，纯 CPU 代码需要主动检查点。

需要为每次请求创建 Par 吗？

通常不需要。ParConfig 和 Par 可以在应用启动时创建并复用；每次调用只传入不同的任务列表和选项。

能不能失败后继续处理其他元素？

框架默认 fail-fast。如果业务需要容错，请在单个任务内部捕获异常，将成功或失败包装成统一结果。

如何开始排查线程池嵌套问题？

开启 livelock detection，并为执行器提供名称映射。框架会在请求结束时报告任务级或执行器级环路。
