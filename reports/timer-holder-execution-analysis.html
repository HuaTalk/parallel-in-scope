范围：仅分析 TimerHolder、调度装饰器、cached task pool 和 JDK ScheduledThreadPoolExecutor。
Runtime anatomy / Java 8

到期不等于立即执行

TimerHolder 接受任意数量的延迟任务。默认 scheduler 有两个 worker，它们只负责等待 deadline 并把到期 action 投递给独立的 cached task pool；取消链不会在 scheduler worker 上长时间执行。

2 + cached 2 个 scheduler worker 负责到期 dispatch；实际 timeout/cancel action 使用独立 cached pool。
01
Configuration

功能类实际创建了什么

Holder 只负责延迟初始化和发布单例。真正的执行边界由 scheduler 核心线程数、延迟队列、dispatch 包装器和 task pool 决定。

private static final class TimerHolder {
    private static final int CORE_POOL_SIZE = 2;
    static final ListeningScheduledExecutorService INSTANCE;

    static {
        ThreadFactory threadFactory = new ThreadFactoryBuilder()
            .setDaemon(true)
            .setNameFormat("Par-Timer-%d")
            .setUncaughtExceptionHandler(...)
            .setPriority(Thread.MAX_PRIORITY)
            .build();

        ScheduledThreadPoolExecutor timerImpl =
            new ScheduledThreadPoolExecutor(
                CORE_POOL_SIZE, threadFactory);

        timerImpl.setRemoveOnCancelPolicy(true);
        INSTANCE = createDispatchingTimer(timerImpl);
    }
}

private static final class TimerTaskExecutorHolder {
    static final ExecutorService INSTANCE = Executors.newCachedThreadPool(
        new ThreadFactoryBuilder()
            .setDaemon(true)
            .setNameFormat("Par-Timer-Task-%d")
            .build());
}

// scheduler worker 只执行这一小段 dispatch
scheduler.schedule(
    () -> TimerTaskExecutorHolder.INSTANCE.execute(action), delay, unit);
两个 scheduler worker

corePoolSize=2 提供两个延迟队列取出与 dispatch 槽位，不再限制取消 action 的并发执行数。

近似无界延迟队列

任务先进入 DelayedWorkQueue。到期后 scheduler 只提交短小的 dispatch，随后立即回到队列。

取消动作隔离

真正的 timeout/cancel 链在线程名 Par-Timer-Task-* 上执行；scheduler worker 不执行用户 action。

02
Submission path

从 schedule 到 action 返回

每次提交先创建延迟触发器。到期时 scheduler worker 只负责把 action 投递到 cached pool，实际取消和 Future listener 不占用 scheduler。

STEP 1 调用 schedule

传入 Runnable、delay 和 TimeUnit；可来自默认 TimerHolder 或用户配置的 timer。

STEP 2 进入延迟队列

scheduler 保存 trigger，按 deadline 排序；相同 deadline 按队列规则竞争。

STEP 3 scheduler dispatch

Par-Timer worker 到期后只调用 cached pool 的 execute()，不运行取消链。

STEP 4 task pool 执行

Par-Timer-Task-* 执行 timeout/cancel action；多个 action 可以同时运行。

STEP 5 完成或取消

action 完成 Future；调用方取消时同时取消 trigger 和尚未开始的 action。

提交数量由 scheduler 的延迟队列承载，不受 core=2 拒绝限制。
调度并发最多两个 scheduler worker 取出到期 trigger 并 dispatch。
action 并发cached pool 按需创建线程；资源耗尽、系统调度或 action 自身仍可能造成延迟。

04
Worker occupancy

scheduler 与 action 的占用边界

未来 deadline 之前 scheduler worker 阻塞在延迟队列；到期后只做 dispatch。取消链、Future 完成和同步 listener 的耗时转移到 task pool。

WAITING
scheduler 等待 deadline

Par-Timer worker 阻塞在 DelayedWorkQueue.take()，这是正常等待，不等于线程泄露。

DISPATCH
取出并投递 action

deadline 到达后，scheduler 只调用 cached pool 的提交接口；这段代码应保持短小。

ACTION
task pool 执行取消链

包括 Future 取消、超时回调和同步 listener。阻塞会占用 Par-Timer-Task-*，不占用 scheduler。

RELEASE
action 返回

cached pool 线程回收或复用；周期任务在 action 完成后安排下一次触发，避免同一任务重叠执行。

两类线程的实际占用时间 scheduler：等待 + dispatch
task pool：取消 action + listener

core=2 提高调度 dispatch 的并发余量；cached pool 仍受内存、线程数和业务 action 时长影响。

05
Boundary

配置能够保证什么

默认 TimerHolder 将“何时触发”和“如何执行取消”分离；用户也可以通过 ParConfig.builder().timer(customTimer) 指定自己的 scheduler。

能够保证

scheduler worker 不执行用户定义的 timeout/cancel action；三个同时到期任务会快速完成 dispatch；默认取消 trigger 会从队列移除；自定义 timer 的生命周期由用户控制。

不能保证

action 恰好在 deadline 执行；cached pool 无限可用；用户自定义 scheduler 自身不被错误配置拖慢；协作式取消能强制终止不响应中断的业务代码。
